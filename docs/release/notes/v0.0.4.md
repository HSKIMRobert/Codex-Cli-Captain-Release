## Draft Documentation / Operator Policy

- tracks `rust/ccc-mcp/src/main.rs` as oversized and in need of careful modularization so the CCC MCP entrypoint stays readable and future maintenance/editing burden stays lower
- treats `rust/ccc-mcp/src/main.rs` modularization as repeated seam-by-seam maintenance work, not a one-off cleanup: each extraction should name the cohesive seam, document the landing, preserve behavior/contracts, and keep focused regression coverage around the moved behavior
- documents Captain-Mediated Review as an explicit and conditional captain-owned policy: review is bounded to verification input and still leaves accept/reassign/close decisions with the captain
- defines resource-aware review gating and bounded reviewer/pass limits so hardware, memory, and same-machine concurrency cost can suppress, reduce, or require review without allowing unbounded review swarms
- defines captain dissatisfaction handling after subagent results: the captain may accept, close, or mark output unsatisfactory; unsatisfactory results should be recorded with rationale and next action; repair stays bounded to one same-specialist pass when scope still fits or one reassignment when the role was wrong
- documents Captain Review Intervention policy: when the captain reviews a result or observes active-work risk and finds the output or direction unsatisfactory, it should record a bounded delta plus rationale in LongWay/task-card state and resolve with exactly one captain action: safe same-worker amend, reclaim, or reassignment
- keeps user guidance during active work as secondary input routed only through the captain, not as a direct user-to-subagent side channel; stale output should stay visible and cannot silently overwrite the chosen path unless the captain explicitly merges it
- notes that reclaim overlap can briefly increase CPU and memory load, so the default should stay single-path and the only persisted artifact should be the intervention delta plus decision record
- clarifies captain execution discipline: host Codex as captain owns LongWay, routing, lifecycle, fan-in, review, validation, and commit boundaries; ordinary read-only investigation, docs edits, code/config mutation, and review judgment should go to `ccc_scout`, `ccc_scribe`, `ccc_raider`, and `ccc_arbiter` via custom subagents when available; direct captain edits stay limited to explicit fallback, trivial operator-side fixes, or recorded CCC degradation
- documents `Too many open files (os error 24)` as a resource-pressure signal that should pause new reviewer/specialist launches until active host agents are terminally recorded, merged or reclaimed, and closed in the host session
- clarifies that host-agent close is only one cleanup path: detached terminal work, abandoned `codex exec` fallbacks, and stale CCC/Codex-owned MCP helper descendants must be surfaced separately and stopped only after operator confirmation
- adds the v0.0.4 cross-platform release requirement: v0.0.3 is treated as macOS-oriented, and v0.0.4 should make Linux and Windows install, packaging, and smoke-test support first-class release criteria instead of macOS-only assumptions
- defines commit-per-work-unit discipline for 0.0.4 work: runtime, docs, release metadata, and test repair changes should be committed as coherent slices instead of one mixed mega-commit unless captain records why combining is safer
- adds release workflow commit-boundary guidance so publishing work separates runtime changes, focused test repairs, source docs, release-repo docs, release metadata, and generated assets unless a combined commit is explicitly justified
- clarifies completion discipline for document-backed work: when the operator asks CCC to continue a referenced plan/release note/checklist to completion, the captain must derive remaining items and continue bounded slices until each in-scope item is completed, explicitly deferred, or blocked with an operator decision

## Planned Completion Scope Before v0.0.4 Release

`v0.0.4` should be treated as complete only when the runtime and docs support:

- explicit captain review gates instead of automatic review on every delegated task
- full resource-aware review limits for hardware, memory, and same-machine concurrency pressure beyond the first persisted suppression decision
- bounded reviewer counts and pass counts, with no free agent-to-agent chat or untracked reviewer chains
- read-only reviewer findings that feed verification state while captain keeps accept, reassign, repair, and close authority
- status/quiet lifecycle output that makes both opened and intentionally skipped review visible
- captain-owned repair/reassignment follow-up now queues through the intervention state, with unavailable or exhausted budget choices and missing reassign targets staying blocked and producing no follow-up task card
- terminal subagent cleanup visibility so closed, stale, or reclaimed worker handles do not silently exhaust host resources
- explicit operator guidance for `Too many open files (os error 24)`: keep work single-path, avoid opening more agents, close every spawned host agent after terminal fan-in, then inspect detached terminal and stale Codex/MCP helper descendants if pressure persists
- config migration visibility so setup/check-install can report current, stale, migrated, preserved, skipped, conflict, backup, dry-run, rollback, and restart-required states for `ccc-config.toml`; packaged `$cap`, MCP registration, and custom-agent expansion remain separate bounded slices

- continuing bounded `rust/ccc-mcp/src/main.rs` extractions, with each seam documented as it lands, behavior and JSON contracts preserved, and regression coverage kept around the moved surface until the entrypoint is manageable
- document/checklist-driven "finish the work" requests keep persisted completion discipline visible across recommendation, auto-entry, status, and task-card surfaces so captain cannot treat a first partial slice as complete

- remaining commit-per-work-unit validation and operator guidance refinements after the initial release workflow guidance
