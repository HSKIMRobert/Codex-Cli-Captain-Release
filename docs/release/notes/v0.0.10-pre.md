# v0.0.10-pre

> Pre-release note: `v0.0.10-pre` focuses on operator-visible CCC truth: clearer LongWay owner rows, routing-drift visibility, CCC-native graph context, opt-in workspace memory, and verified long-session rollover guidance.

## What Changed

- LongWay rows keep their checklist shape while exposing optional planned owner/role metadata and compact lifecycle sync when backed by task-card lifecycle truth.
- `ccc status`, `ccc activity`, fan-in records, and compact checklist output now keep checklist progress separate from subagent lifecycle truth instead of implying one checklist row per spawned host subagent.
- The repair publish set adds nested lane lines under each LongWay task row, short graph-reference messages, local `ccc ...` / shell command wording for operator-visible lifecycle evidence where CCC controls the text, and `captain_instruction` memory status/schema support outside hardcoded prompts.
- Way plans the complete LongWay before dispatch, and each row maps to a subagent or task unit so captain can route the work cleanly.
- Long-session rollover guidance is verified for context, token, and host-subagent pressure. CCC surfaces checkpoint/resume guidance while `/compact`, `/new`, and `/exit` remain operator choices.
- Run lookup and closeout paths now fall back to central workspace storage when a cwd hint misses a known run id.
- Assignment-quality routing drift is surfaced in status when the current task card's owner does not match the inferred specialist family, including docs/planning routed away from `ccc_scribe`.
- The CCC-native Rust graph path is now a core surface through `ccc graph` and `ccc_code_graph`, covering persistent graph storage, incremental indexing, blast-radius/review context, callers/callees/imports/tests, flow tracing, criticality, architecture overview, full-text search, and multi-repo search.
- Opt-in CCC memory is documented and implemented as a small workspace file with explicit `ccc memory status`, `preview`, `write`, and `off` actions.
- Memory writes require preview/diff and stale-write protection. Memory accepts only user preferences, repeated rules, and verified project facts; LongWay state, run state, latest work results, and inference-only observations are rejected as memory truth.
- `ccc memory` is the always-read, compact, persisted captain guidance surface. `$cap` remains the invocation and routing path, not the memory store.
- Way-driven multi-row LongWay planning is implemented with persisted planned rows, public `ccc start` / `ccc run` input support, and captain just-in-time materialization through `ccc orchestrate` advance.
- Graph-assisted routing evidence can be carried as bounded planned-row/task-card metadata. Raw graph dumps are filtered from persisted/status surfaces, and graph evidence is not automatically injected into prompts. When graph evidence drives a visible plan, the output should say so explicitly instead of omitting the graph signal.
- The generated `ccc-config.toml` stays focused on agent settings plus linked feature settings; broader policy stays in runtime state or docs instead of being dumped into config.
- `$cap` and managed prompt surfaces stay on compact CLI/status/checklist contracts to reduce token load; lifecycle fan-in passes only accepted interpretation, bounded scope, evidence paths, and acceptance checks.
- Captain clarification is bounded: broad, risky, ambiguous, or irreversible work asks 1-3 high-signal questions before execution, while narrow work proceeds with explicit assumptions.
- Captain instruction memory is separate from hardcoded prompts at the memory/status/schema layer, so recurring clarification obligations can be stored as `captain_instruction` entries for later captain use instead of living only in prompt text.
- Captain/orchestrator remains the host control-plane actor and is not packaged or synced as a normal spawnable custom-agent lane.

## Validation Evidence

Local pre-release validation completed during the 0.0.10 work stream:

- Focused long-session, run-locator, lifecycle-sync, graph, memory, and assignment-quality tests passed individually.
- Publish-stage `cargo test -p ccc` passed with `213 passed; 0 failed` after the LongWay planned-row, JIT materialization, and bounded routing-evidence work.

The release publish pass reruns source tests and release packaging checks after this note and the `0.0.10-pre` metadata are applied.

The repair publish set is additive. Rollback and downgrade continue to use the original `v0.0.10-pre` release assets, which remain available unchanged.

## Deferred

- Goal-mode behavior and anything depending on upstream goal-mode interception remains out of this train.
- Automatic memory writes, cross-workspace memory, semantic search/embeddings, external memory services, and automatic prompt injection are deferred.
- Deeper automatic graph consumption by review/fan-in remains deferred; explicit CLI/MCP graph queries are the shipped surface.
