# v0.0.13-pre

> Pre-release note: `v0.0.13-pre` promotes the 0.0.12 optional SSL sidecar experiment into a bounded Skill Registry evidence layer. `$cap` remains the public entrypoint; persisted CCC run state, LongWay, task cards, checklist, fan-in, and status remain runtime truth.

## What Changed

- Added a modular Skill Registry wrapper over `SKILL.md`, SSL manifests, and role config evidence.
- Added conservative SSL manifests for all managed custom agents: `ccc_tactician`, `ccc_scout`, `ccc_raider`, `ccc_scribe`, `ccc_arbiter`, `ccc_sentinel`, `ccc_companion_reader`, and `ccc_companion_operator`.
- Hardened SSL manifest validation with required `version`, `skill_id`, Scheduling, Structural, and Logical fields. Missing, invalid, stale, or drifted manifests remain non-blocking evidence.
- Connected Router shortlist output to registry Scheduling metadata so selected agent/model/variant/reasoning values can cite their display source.
- Connected Sentinel and Arbiter review precheck output to registry Logical metadata so risk classification and high-risk review ownership can cite registry evidence.
- Connected Way `PLAN_SEQUENCE` planning context to registry Structural scenes from `ccc_tactician`.
- Added a bounded Way clarification interview loop for broad or ambiguous `PLAN_SEQUENCE` work. CCC can persist 1-3 questions, wait for the operator, consume the answer exactly once, and regenerate pending LongWay rows.
- Added registry health visibility to `ccc check-install`, including available/non-available counts for managed-agent manifests.
- Improved App/CLI visibility: planned rows now show separate agent/model/reasoning source labels, target-root warnings include a structured retry command, and the pre-release plan documents direct `$cap`, CCC-owned `PLAN_SEQUENCE`, app-panel text, checklist text, and quiet CLI smoke paths.
- Documented the quiet `ran` lifecycle contract for operator-visible CCC mutations: `ccc start`, `ccc orchestrate`, `ccc subagent-update`, `ccc memory`, `ccc checklist`, and `ccc status` should prefer `--quiet --json` inline payloads, while `status --text` stays the compact transcript path.
- Clarified that configured `ccc_*` custom agents are the default specialist targets, that `ccc memory` is opt-in and unconfigured by default, and that mutation completion still waits for specialist fan-in plus arbiter review where review-sensitive changes need it.
- Updated release asset staging and matrix verification so packaged assets include `skills/ssl/*.skill.ssl.json` and fail smoke checks if a managed-agent manifest is missing or has the wrong `skill_id`.
- Refreshed generated defaults so the legacy `planner` alias and Way/tactician custom agent sync use the documented `gpt-5.5` high-reasoning setting.
- Tightened the follow-up release path: inline `--quiet --json` is the default operator-visible payload transport, missing `subagent-update` task cards report the task-card id/path, already-passed verification does not reopen the same review gate, and `host_subagent_thread_limit` is now a structured fallback reason.
- Hardened planned-row routing and display: git/gh mutation rows route to `companion_operator` before docs heuristics, bare words such as `stage` no longer match `tag`, and planned rows no longer inherit current-task agent/model data unless their `task_card_id` matches.
- Updated the operator-facing release README set and kept internal 0.0.13-pre follow-up notes out of the release repository.
- Follow-up fixes landed for the refreshed release set: README variants no longer carry internal lifecycle transcript guidance; lifecycle diagnostic routing no longer treats the `subagent-update` command token as mutation by itself, while explicit mutate/mutation/repair language still routes mutation; installed CLI smoke initial task routes scout/read-only without assignment drift.
- Follow-up release-card fixes landed: ordinary LongWay/checklist transcript output is plain and compact so LongWay rows are not hidden by folded boxed app-panel output, and README support wording now states the exact target matrix with macOS normally supported and Linux/Windows available with possible platform-specific issues.
- Follow-up captain-boundary fixes landed: when host subagent capacity is exhausted, captain should wait for fan-in, close completed host agent threads, or record reclaim/reassign/terminal fallback before retrying specialist dispatch; direct specialist-owned mutation remains blocked by default.
- Follow-up operator-visibility fixes landed: `ccc status/checklist --subagents --text` renders a short subagent-lane-only view, and `ccc status/checklist --projection` writes one workspace-root `CCC_LONGWAY_PROJECTION.md` display artifact for `git diff` review. Projection output is capped, lane-based, and language-aware for Korean operator requests.
- Follow-up host-close visibility landed: terminal host-subagent updates release CCC's active handle and status surfaces when the host still needs to close the completed agent thread.
- Follow-up Way-first integration landed: new `$cap`/CCC runs default to `PLAN_SEQUENCE` and auto-sync `CCC_LONGWAY_PROJECTION.md` unless the operator explicitly skips planning/LongWay or resumes an already-approved LongWay. Operator approval moves the run forward with `approve_longway=true`, so captain/subagent progress remains visible through the projection file and Codex diff.
- Follow-up entry-surface fixes landed: runtime entry policy treats legacy `codex_cli_foreman_first` as the canonical `codex_cli_ccc_first` behavior, while `check-install` flags legacy or invalid `entry_policy.mode` values for setup/backfill instead of reporting the config fully current. The CLI now exposes `ccc recommend-entry` and `ccc auto-entry` with JSON output by default; CLI auto-entry refreshes `CCC_LONGWAY_PROJECTION.md` after creating a run.
- Follow-up prompt hygiene landed: packaged `$cap` skill text, role developer instructions, worker execution prompts, and generated runtime prompt caps were tightened to reduce repeated token load while preserving CCC ownership, mutation guardrails, fan-in, and Korean operator-facing behavior.

## Scope Notes

`v0.0.13-pre` still treats SSL manifests as advisory evidence. The registry is a lookup and visibility layer, not a replacement for approved LongWay state, task cards, fan-in, or role config.

`$cap` is the CCC entrypoint. Host `/plan` and `/goal` should not be documented as CCC orchestration paths, and host Plan Mode cannot be triggered inside Way as a background Codex Plan Mode engine. CCC `PLAN_SEQUENCE` and the configured Way agent own planning; host proposed-plan output must not replace CCC LongWay, task cards, checklist, fan-in, or status truth.

Operator-visible lifecycle output should stay compact. Prefer quiet `ran` commands for mutations, keep `ccc checklist`/`ccc status` rows concise, and reserve verbose JSON transcript output for non-operator paths or explicit debugging.

New operator work should normally start with the Way-first LongWay projection flow. Way creates the plan, captain materializes and refreshes `CCC_LONGWAY_PROJECTION.md`, and Codex diff shows that projection. `CCC_LONGWAY_PROJECTION.md` is the primary operator-visible progress artifact for Codex diff review; persisted run state, task cards, lifecycle, and fan-in remain truth. Once the projection/diff flow is active, status no longer needs to embed LongWay details and standalone checklist transcript output is not required for normal progress visibility.

Configured `ccc_*` custom agents are the default specialist names in the release flow. Generic `worker` and `explorer` labels do not apply unless an explicit override says otherwise.

`ccc memory` is opt-in and unconfigured by default. The SSL Skill Registry is available as bounded evidence for routing, planning, and review, but it does not replace persisted run state.

Mutation completion still waits for specialist fan-in, and arbiter review remains the final gate for review-sensitive changes.

## Follow-Up Improvements

- Generate explicit Way planned-row role/agent metadata earlier so the
  materializer does not need to infer read-only evidence routing from text.
- Tighten stalled scout retry/reassign behavior and make arbiter-over-raider
  review handoff appear as verification, not policy drift.
- Generalize stalled host custom subagent recovery across scout, raider, and
  arbiter so a missing fan-in envelope triggers bounded retry/reassign instead
  of manual degraded companion fallback.
- Keep the worker-harness recovery path visible in status/checklist output so
  operators can distinguish primary host-subagent fan-in from codex exec
  recovery fan-in.
- Prefer explicit LongWay planned-row owner metadata in assignment-quality
  checks before smoke/status diagnostic heuristics, so intentionally planned
  raider or arbiter rows are not reported as routing drift.
- Remove README/release/packaged-skill wording that presents host `/plan` or
  `/goal` as CCC orchestration paths. `$cap` should be documented as the entry
  point, and Way should be described as CCC `PLAN_SEQUENCE`, not background
  host Plan Mode.
