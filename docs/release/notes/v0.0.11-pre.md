# v0.0.11-pre

> Pre-release note: `v0.0.11-pre` starts the CCC Sisyphus walking-skeleton path. `$cap` remains the public entrypoint, while CCC owns LongWay, task cards, checklist, fan-in, persisted status, and restart handoff without depending on host `/plan` or `/goal`.

## What Changed

- Added the Loop 1-4 CCC walking skeleton for Intent Frame, Way planning, pending LongWay, task cards, checklist, scheduler/router seams, compact fan-in, context health, and restart handoff.
- Split the intended harness into `PLAN_SEQUENCE` and `EXECUTE_SEQUENCE`: planning produces a pending LongWay, and execution starts only after operator approval.
- Clarified that host `/plan` is an optional planning UI or mode and does not replace CCC `PLAN_SEQUENCE`.
- Clarified that host `/goal` is an optional outer objective hint and does not replace CCC LongWay, checklist, fan-in, status, or completion gates.
- Kept public `skills/cap/SKILL.md` thin. Internal routing, lifecycle, fan-in, fallback, context, and compatibility policy belongs in `docs/release-work/0.0.11/CCC_MEMORY.md` or persisted `captain_instruction` guidance.
- Added memory captain-instruction source visibility so durable captain guidance can be surfaced without treating memory as task state.
- Added companion runtime role defaults while keeping captain as the control-plane actor, not a normal spawnable specialist lane.
- Aligned graph wording with Way planning and bounded graph evidence usage.
- Projected planned LongWay rows under matching status phases while keeping `longway.planned_rows` as canonical truth. `phase_rows[].planned_rows` remains status/checklist projection only.
- Added `ccc status --app-panel` as a transcript-readable Codex app fallback surface when the host app does not render a persistent side panel.
- Added a render-oriented `ccc_render_app_panel` MCP tool plus
  `ui://ccc/app-panel.html` resource metadata for MCP Apps hosts that can render
  result components, and added `ccc status --app-panel --artifact` to write
  `CCC_LONGWAY_PANEL.md`/`.json` run artifacts when native panel rendering is
  unavailable.
- Added near-native Codex App fallback artifacts by keeping run-local
  `CCC_LONGWAY_PANEL.md`/`.json` and also updating workspace-level
  `CCC_LATEST_PANEL.md`/`.json` so the latest CCC LongWay/checklist/subagent
  dashboard is easier to find from the app artifact surface.
- Reworked app-panel Markdown and transcript text into a boxed dashboard that
  shows run state, LongWay checklist rows, planned rows, current task
  role/agent/model/variant, subagent activity with next action, and
  non-blocking warnings in the normal Codex App/CLI output area.
- Added MCP Apps `ui.prefersBorder` metadata for hosts that render the
  `ui://ccc/app-panel.html` component, while keeping the boxed transcript as the
  guaranteed fallback when native right-panel rendering is host-owned.
- Added visibility diagnostic routing so smoke/status/install/app visibility checks stay read-only and do not create review-gate or assignment-drift warnings by themselves.

## Important Scope Note

`v0.0.11-pre` is not full CCC runtime parity and not a completed rebuild. It is a walking skeleton plus follow-up hardening base.

Deliberately deferred:

- Loop 5 Intervention / Review-retry-replan-reclaim parity projection
- fixed schema contract promotion for `ccc_loop.additionalProperties=true`
- setup_config config surface change
- full runtime parity projection

Intervention and review/retry/replan/reclaim behavior remains primarily in the existing lifecycle, `subagent_update`, `orchestrate`, and status paths. The next phase should review those paths as parity projections instead of introducing a second source of truth.
