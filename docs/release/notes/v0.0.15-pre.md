# v0.0.15-pre

`v0.0.15-pre` is the current public pre-release note for the Rust-owned CCC
harness. It carries forward the `0.0.14-pre` intent-state-machine work and
focuses the release on checkpoint/resume clarity, background lifecycle
visibility, verification gates, plugin packaging, and release-validation
discipline.

`$cap` remains the public CCC entrypoint. The packaged CCC plugin files are for
installation and discovery, not a replacement operator command surface.
The release installs through the local `ccc-local` Codex plugin marketplace and
removes the legacy direct `mcp_servers.ccc` registration plus any standalone
`~/.codex/skills/cap` copy.

## What Changed

- Bumped source runtime and release metadata to `0.0.15-pre`.
- Kept `.codex-plugin/plugin.json` versioned as `0.0.15-pre` for the plugin
  package surface.
- Documented the plugin package as a distribution surface for `.mcp.json`,
  `skills/ccc/SKILL.md`, and the bundled `$cap` flow.
- Added display-only StarCraft-style callsigns while preserving stable `ccc_*`
  IDs as routing truth. Operator-facing status/projection text now shows the
  callsign plus stable ID where feasible, while machine-readable payload fields
  keep stable IDs. The README tables now make the display-only callsign role
  explicit by separating stable agent ID, display callsign, model, reasoning,
  and notes:
  - `ccc_tactician` -> Executor
  - `ccc_scout` -> Observer
  - `ccc_raider` -> Marauder
  - `ccc_scribe` -> Adjutant
  - `ccc_arbiter` -> Arbiter
  - `ccc_sentinel` -> Overseer
  - `ccc_companion_reader` -> Probe
  - `ccc_companion_operator` -> SCV
- Clarified that host UI layers may emit outer notifications such as
  `Closed Carver [ccc_scout]`, but that wording is host-managed and not a CCC
  guarantee; CCC-controlled status/projection surfaces use callsign-plus-
  stable-ID forms such as `Observer(ccc_scout)`.
- Added advisory workflow metadata inspired by `oh-my-openagent`:
  `github-triage`, `get-unpublished-changes`, `remove-deadcode`,
  `ai-slop-remover`, `lsp-safe-refactor`, `review-work`,
  `pre-publish-review`, `hyperplan`, `git-master`, `publish`,
  `release-command-discipline`, `release-note`, `readme-maintenance`,
  `changelog`, `role-ownership`, `lane-conflict`,
  `fallback-classification`, and `filesystem-evidence`.
- Added advisory per-agent metadata in generated config and SSL manifests:
  `display_name`, `callsign`, `theme`, `inspired_by`,
  `recommended_workflows`, and `lsp_capabilities`.
- Completed the Sentinel/schema repair in runtime/config/status surfaces:
  the generated config schema declares `agents.sentinel/defaults`, run and
  task-card schemas persist `sentinel_intervention` fields/history/latest
  entry trace, and `subagent-update --text` now renders a compact Sentinel
  line.
- Added a bounded TypeScript/JavaScript LSP config/schema surface for
  `lsp_diagnostics`, `lsp_references`, `lsp_definition`,
  `lsp_prepare_rename`, and `lsp_rename` using
  `typescript-language-server --stdio`. Runtime LSP execution is explicitly
  deferred in `0.0.15-pre`; CCC does not start language servers yet. The
  bounded design note stays separate from Sentinel guardrails and prompt
  refinement, and lives at
  `docs/release-work/0.0.15/LSP_MVP_DEFERRED.md`.
- Documented the implemented, config-gated Graphify-backed Graph Context
  Provider as the routing shim behind the existing public `ccc graph` /
  `ccc_code_graph` surfaces. The docs now say graph context is default-off,
  existing graph-facing surfaces stay stable, enabled-and-ready routing uses
  Graphify-backed provider evidence, enabled missing/stale/unavailable
  Graphify falls back to normal scout/source evidence, full `graph.json` is
  not loaded by default, and no new public graph command is added.
- Documented optional local LSP setup with
  `npm install -g typescript typescript-language-server` for TypeScript and
  JavaScript, and `rustup component add rust-analyzer` for Rust-only local
  navigation.
- Clarified that optional setup guidance and install checks can surface
  TypeScript/JavaScript LSP and Rust LSP readiness, but `0.0.15-pre` still
  defers runtime LSP execution. `rust-analyzer` remains an optional Rust-only
  local-navigation surface unless future docs explicitly say otherwise.
- Added a future-work item for a later runtime Rust LSP bridge: process/session
  management for `rust-analyzer`, request wrappers for diagnostics,
  definition, references, prepareRename, and rename, setup and check-install
  readiness coverage, safe rollback/fallback when `rust-analyzer` is missing,
  and focused tests/acceptance. `0.0.15-pre` itself remains metadata-only and
  does not start language servers.
- Documented optional future agents without enabling runtime roles:
  `ccc_release_arbiter`/Judicator, `ccc_qa_runner`/Valkyrie, and
  `ccc_lsp_scout`/Science Vessel.
- Updated install-surface checks so packaged `$cap` skill inspection resolves
  the plugin-cache install path used by `ccc@ccc-local`.
- Updated `check-install` wording so optional surfaces read as setup/readiness
  checks rather than runtime activation, including the deferred Rust LSP
  surface.
- Updated install-surface registration matching so the versioned
  `ccc@ccc-local` plugin-cache MCP binary is accepted as current while wrong
  versions, commands, or args still report a stale registration.
- Tightened the packaged `$cap` dispatch contract: creating or resuming a CCC
  run is not enough; executable `current_task` routing must immediately hand
  configured `ccc_*` or companion-owned work to the assigned host custom agent
  and record `subagent-update` fan-in events.
- Restored the natural CCC workflow-loop projection that operators associated
  with `0.0.14-pre`: requirements understanding -> planning -> exploration ->
  modification -> review -> verification now appears as a compact
  `workflow_loop` surface in status/projection, compact status, and app-panel
  payloads. This is visibility over persisted LongWay/task-card truth and does
  not add public commands or alter specialist routing.
- Documented future work, not implemented in `0.0.15-pre`, for preferring
  StarCraft-style display callsigns such as `[Marauder]` and `[Observer]` for
  operator-visible bracket labels like `[ccc_raider]` and `[ccc_scout]` where
  CCC can influence the presentation. Stable `ccc_*` IDs remain the
  machine-readable routing truth; if these display changes are implemented
  later, the source README and release README surfaces may also need updates so
  user-facing docs match the new output.
- Documented future work, not implemented in `0.0.15-pre`, for preferring the
  compact `Ran ccc check-install --text` framing over the full absolute-path
  form in host-visible run logs when CCC can influence the presentation.
  Stable `ccc_*` IDs remain unchanged, and any later change needs acceptance
  that separates CCC-controlled log text from host framing.
- Documented future work, not implemented in `0.0.15-pre`, to improve the
  routing classifier when the only target is documentation or release notes so
  docs-only work routes to `Adjutant(ccc_scribe)` rather than
  `Marauder(ccc_raider)`. This routing change is separate from the display
  label and run-log wording work. Stable `ccc_*` IDs stay unchanged, and the
  future acceptance should prove docs/release-note-only prompts do not drift
  into the mutation path.
- Added the first runtime plumbing slice for an internal
  `ccc_promptsmith`/Ghost prompt-refinement role: the config-only gate is
  wired so generated defaults version 16 sets
  `[features].prompt_refinement = false`, bootstrap uses config-derived
  enabled state, `ccc.prompt_refinement.v1` metadata/status is persisted in
  `run.json`, and status payload/text surface it while the feature stays
  default-off and non-executing. The current slice also records an internal
  `prompt_refinement_handoff_decision` envelope plus Captain-owned gating
  fields such as `owner=captain`,
  `captain_gate=accept_adjust_reject`, `longway_materialization_allowed=false`,
  and `task_card_creation_allowed=false`. The envelope records pending
  Captain accept/adjust/reject semantics but does not execute them. Actual
  Ghost/model dispatch, refined-brief materialization, and downstream
  LongWay/task-card creation are still future work.
- Clarified that Ghost remains internal, compact English-only, and
  non-executing for now; it does not replace Captain, directly create
  LongWay/task cards, directly close agents, or change final user-facing
  response language.
- Documented Goal Bridge as opt-in internal `$cap` machinery with
  `[features].goals` defaulting to `false`, the `[goal_bridge]`
  config/schema/default shape, generated defaults version 16, focused tests,
  and a minimal internal non-executing Captain-owned
  `ccc.goal_bridge.v1` run record when enabled. Host goal state remains a
  secondary host aid, never CCC truth, and role-specific subgoal context,
  fan-in goal annotations, host goal capability integration, and host goal
  mutation remain future/runtime-gated/not implemented.
- Documented a future/internal Ghost brief detail that may include idle or
  unfinished host-subagent observations in the compact internal brief so
  Captain can make the normal closure decision and action. Ghost still does
  not directly close agents or replace Captain ownership.
- Documented three additional future-work themes, not implemented in
  `0.0.15-pre`, to keep later work visible:
  - Hooks-based guardrails with `UserPromptSubmit`, `PreToolUse`,
    `PostToolUse`, and `Stop` style lifecycle hooks to prevent routing drift,
    unsafe commands, missed verification, missing `$cap`/CCC loop behavior,
    and Captain-local execution of specialist-owned work. The guardrail shape
    is a run-scoped `ccc_sentinel`/Overseer layer inside `$cap`, not an
    always-running host subagent, public skill, or public command.
  - Compact evidence/output that compresses validation and tool output while
    preserving raw artifacts and log paths for audit. This is RTK-inspired
    discipline, not a commitment to adopt RTK wholesale.
  - App-server/remote status exposure that strengthens CCC status,
    projection, fan-in, and release-validation surfaces for app-server and
    remote workflows.
- Documented agent parallelization as a `0.0.15-pre` work item rather than a
  staged check-first-then-later plan. The intended release scope is the full
  set of parallel-task decomposer/task graph, read-only fan-out, path conflict
  guard, fan-in aggregator, Sentinel routing, Captain-local-work guardrail,
  lane-conflict classification, Codex hooks guardrail integration, worktree
  isolation for parallel mutation, parallel and specialized review lanes,
  app-server/remote visibility for parallel state, and focused
  tests/acceptance, while keeping stable `ccc_*` role IDs and
  StarCraft-style callsigns consistent.
- Aligned README, install guidance, release docs, validation runbook, and
  release verification scripts with the `0.0.15-pre` source version.
- Preserved the release workflow boundary: public GitHub release cards, assets,
  tags, pushes, and release-repo edits happen outside this source-prep note.
