# v0.0.5-pre

> Pre-release notice: `v0.0.5-pre` is still a beta/pre-release train. Install and restart Codex CLI before evaluating the refreshed `$cap` skill and custom agents.
> Platform status: macOS is officially supported, with macOS arm64 locally tested. Linux and Windows install/run paths are included for early testing, but they have not yet been fully verified on real Linux or Windows environments.

## Theme

`v0.0.5-pre` focuses on captain-constrained harness behavior.

CCC should keep host Codex as the public captain for routing quality, but move more authority into CCC state transitions:

- no-mutation preflight before default run creation
- MCP as a supported control-plane surface
- compact status as the allowed-action contract
- captain action guard for direct mutation, finish, review skip, and fallback
- mandatory fan-in before finish or follow-up
- visible direct-fallback and degraded-path history
- continued reduction of `$cap` skill text toward bootstrap guidance, with core harness policy moving into runtime/MCP state instead of long public prompt text

## Runtime Slices Already Landed

- `ccc_recommend_entry` now serves as the no-mutation preflight gate for default run creation.
- `ccc_status`, `ccc_orchestrate`, and `ccc_subagent_update` are treated as supported control-plane surfaces for persisted state transitions and fan-in.
- The installed `$cap` skill is being reduced toward bootstrap guidance; durable routing, fallback, fan-in, review, and repair policy belong in runtime/MCP-enforced state as those guards land.
- The source README translations and release-work plan now mirror the same 0.0.5 control-plane wording.

## Runtime Scope

- `ccc_recommend_entry` is the no-mutation preflight gate for default run creation.
- MCP status/orchestration/lifecycle calls are documented as supported control-plane usage.
- Status now exposes the captain action contract so the host captain can see the allowed action, required action, preflight guard, and denial reason.
- Review-gate detection blocks direct captain action for required and recommended review states.
- Unsatisfactory/needs-work fan-in now canonicalizes to bounded specialist follow-ups, and captain-side local repair stays out of the path when CCC can route repair or reassignment through a specialist.
- Specialist fan-in and visible fallback policy remain captain-owned and bounded.
- Durable routing, fallback, fan-in, review, and repair policy continue moving out of long `$cap` prompt text and into runtime/control-plane enforcement.
