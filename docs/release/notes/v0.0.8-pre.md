# v0.0.8-pre

> Pre-release note: `v0.0.8-pre` tightens operator guidance for comment and annotation blocks, harness-style workflow framing, subagent stall recovery, LongWay owner identity rows, release/install mutation routing, and token-aware skill/agent prompt discipline.

## Theme

`0.0.8-pre` combines operator-facing guidance with bounded runtime hardening. It keeps the existing captain-first CCC shape, while making the rules and enforcement easier to follow when the request is about comments, annotations, risky intent confirmation, recovery after a stalled subagent, LongWay owner visibility, release/install repair routing, or token-aware specialist prompting.

## Operator Guidance

- When the operator asks for comments or annotations, preserve the requested chronological block format instead of flattening or reordering the content.
- Treat OMO sisyphus or harness wording as CCC's operating shape: one captain, bounded specialist routing, and every specialist result returns to captain before the next decision.
- For complex or risky interpretations, captain should confirm the reading with the operator before handing accepted work to Way/tactician.
- Keep role separation explicit: scout and companion_reader gather evidence, Way/tactician plans, raider and companion_operator mutate, scribe handles docs or operator text, arbiter reviews risk or acceptance, and captain owns fan-in.
- If a routed subagent stalls or disappears before fan-in, prefer reclaim, retry, or reassign before degraded captain fallback when delegation is still viable.
- LongWay rows may show optional owner identity when it helps the operator follow the work, for example `[ ] Mill [ccc_scribe] : Clarify 0.0.8 OMO harness-style requirements`.
- Release asset packaging, `install.sh`/`install.ps1` repair, and `gh release upload`/`gh release edit` stay routed through the correct specialist or operator role first.
- Release/update work preserves a downgrade path: previous pre-release assets stay installable, `CCC_VERSION` override paths remain supported, and any public-default rollback is recorded as completed, skipped, or blocked.
- Broad multi-surface work should split into bounded lanes instead of giving one agent rollback, runtime implementation, docs, packaging, upload, and review all at once; captain remains the fan-in and merge point.
- `$cap` should use compact lifecycle surfaces, show the standalone `ccc checklist --text` LongWay block when lifecycle state changed, and avoid passing full status/task-card dumps into specialist prompts.
- `ccc_subagent_update` now supports compact-ref MCP calls: pass `event_ref` and `mode: "compact"` to get a summary-only response while full long fan-in is persisted as a run-local temp artifact.
- Generated custom-agent prompts carry shared token-discipline guidance: use task-specific context, avoid full-history dumps, do not restate unchanged task-card text, and return short structured fan-in.

### Chronological Comment And Annotation Templates

Use the requested block format as a chronology constraint, not as a signal to summarize or reorder the work.

```text
#===============================
# {yy-mm-dd}-{git 기준 작업한 사람 이름 or 닉네임} : {task title}
# {yy-mm-dd}-{git 기준 작업한 사람 이름 or 닉네임} : {task title}
# 시간 순으로 간단하게 요약
#================================
```

```text
#===============================
# {yy-mm-dd}-{git 기준 작업한 사람 이름 or 닉네임}-{task title}
# - {task 내용 너무 길지 않게}
#
# {yy-mm-dd}-{git 기준 작업한 사람 이름 or 닉네임}-{task title}
# - {task 내용 너무 길지 않게}
#================================
```

For code-local comments, place the block directly above the modified function or newly added code. Repeat entries in chronological order when the same task receives additional edits.
